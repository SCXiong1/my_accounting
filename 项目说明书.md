## 一、项目概览

### 1.1 这是什么

一个**移动端个人支出记账 Web 应用**，部署在自己的服务器/NAS 上，通过手机浏览器访问。功能覆盖：记录每笔花销、自定义分类和标签、多维度统计分析消费习惯。

- 核心定位：**轻量、移动优先、个人/家庭支出追踪工具**
- 用户：自己和老婆两人使用，各自独立账号
- 目标：快速记账、清晰分析，不搞复杂功能

### 1.2 技术栈

| 层 | 选型 | 理由 |
|----|------|------|

## 二、需求规格

### 2.1 用户画像

想掌控自己日常开支的个人/小家庭用户，有一定技术能力（能部署个 Python 应用），对数据隐私在意。使用场景：日常随手记一笔花销、月底看看分项统计、年底回顾全年消费。

### 2.2 功能需求（分三阶段）

**P0 - 核心必须：**
- 用户注册/登录（JWT 认证）
- 支出分类 CRUD（预设默认分类 + 自定义），单层扁平结构
- 自定义标签管理
- 支出记录 CRUD：金额、分类、时间、备注、标签（每笔可打多个标签）
- 支出列表查询：按时间范围、分类、标签、关键词筛选 + 游标分页

**P1 - 好用必须：**
- 首页概览：今日/本周/本月/今年支出汇总
- 统计图表：
  - 按分类支出占比（饼图），支持按时间范围 + 标签筛选
  - 按标签支出占比（饼图），支持按时间范围 + 分类筛选
  - 月度趋势（柱状图/折线图），支持按分类 + 标签筛选
- 数据导出 CSV

**P2 - 锦上添花：**
- PWA 离线缓存
- 支出预算提醒
- 数据备份/恢复

### 2.3 非功能需求

| 维度 | 要求 |
|------|------|
| 性能 | 树莓派 1GB 内存流畅运行，API 响应 < 200ms |
| 安全性 | 密码 bcrypt 哈希、JWT 过期、用户数据隔离 |
| 移动体验 | 移动端 PWA，触控友好，可添加到桌面 |

| 数据隐私 | 所有数据本地存储 |

## 三、数据模型设计

> 金额以「分」为单位用整数存储（int64），时间用 Unix 时间戳（秒），软删除用 `deleted` 标记 + `deleted_at` 时间戳。

### 3.1 用户表 (user)

```sql
CREATE TABLE user (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    username        VARCHAR(32)  NOT NULL UNIQUE,
    password        VARCHAR(60)  NOT NULL,           -- bcrypt 哈希
    nickname        VARCHAR(64)  NOT NULL DEFAULT '',
    created_at      INTEGER NOT NULL,
    updated_at      INTEGER NOT NULL
);
```

> 注：SQLite 使用 INTEGER 统一存储整数。SQLAlchemy 模型中所有整数字段使用 `Integer` 类型（非 BigInteger/SmallInteger），以确保 SQLite 的 AUTOINCREMENT 正常工作。

### 3.2 支出分类表 (expense_category)

```sql
CREATE TABLE expense_category (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    uid             INTEGER NOT NULL,                 -- 所属用户
    name            VARCHAR(32) NOT NULL,            -- 分类名称
    icon            VARCHAR(10) NOT NULL DEFAULT '📦', -- 图标（emoji）
    color           VARCHAR(7)  NOT NULL DEFAULT '#607D8B', -- 颜色
    display_order   INTEGER NOT NULL DEFAULT 0,
    deleted         INTEGER NOT NULL DEFAULT 0,
    created_at      INTEGER NOT NULL,
    updated_at      INTEGER NOT NULL,
    deleted_at      INTEGER DEFAULT 0,

    FOREIGN KEY (uid) REFERENCES user(id)
);
CREATE INDEX idx_category_uid_deleted ON expense_category(uid, deleted, display_order);
```

预设分类数据（注册时自动创建）：
| 名称 | icon | color |
|------|------|-------|
| 餐饮 | 🍽️ | #FF5722 |
| 交通 | 🚗 | #2196F3 |
| 购物 | 🛒 | #FF9800 |
| 住房 | 🏠 | #795548 |
| 娱乐 | 🎮 | #9C27B0 |
| 医疗 | 💊 | #4CAF50 |
| 教育 | 📚 | #009688 |
| 其他 | 📦 | #607D8B |

### 3.3 标签表 (expense_tag)

```sql
CREATE TABLE expense_tag (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    uid             INTEGER NOT NULL,
    name            VARCHAR(32) NOT NULL,
    display_order   INTEGER NOT NULL DEFAULT 0,
    deleted         INTEGER NOT NULL DEFAULT 0,
    created_at      INTEGER NOT NULL,
    updated_at      INTEGER NOT NULL,
    deleted_at      INTEGER DEFAULT 0,

    FOREIGN KEY (uid) REFERENCES user(id)
);
CREATE INDEX idx_tag_uid_deleted ON expense_tag(uid, deleted, display_order);
```

### 3.4 支出表 (expense)

```sql
CREATE TABLE expense (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    uid             INTEGER NOT NULL,
    amount          INTEGER NOT NULL,                 -- 金额（分）
    category_id     INTEGER NOT NULL,                 -- 分类 ID
    transaction_time INTEGER NOT NULL,                -- 交易时间（Unix 时间戳）
    timezone_offset INTEGER NOT NULL DEFAULT 480,     -- 时区偏移（分钟，默认 UTC+8）
    note            VARCHAR(255) NOT NULL DEFAULT '', -- 备注
    deleted         INTEGER NOT NULL DEFAULT 0,
    created_at      INTEGER NOT NULL,
    updated_at      INTEGER NOT NULL,
    deleted_at      INTEGER DEFAULT 0,

    FOREIGN KEY (uid) REFERENCES user(id),
    FOREIGN KEY (category_id) REFERENCES expense_category(id)
);

CREATE INDEX idx_expense_uid_deleted_time ON expense(uid, deleted, transaction_time);
CREATE INDEX idx_expense_uid_deleted_category ON expense(uid, deleted, category_id, transaction_time);
```

### 3.5 支出标签关联表 (expense_tag_index)

多对多关联，一笔支出可以有多个标签。

```sql
CREATE TABLE expense_tag_index (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    uid             INTEGER NOT NULL,
    expense_id      INTEGER NOT NULL,
    tag_id          INTEGER NOT NULL,
    created_at      INTEGER NOT NULL,

    FOREIGN KEY (uid) REFERENCES user(id),
    FOREIGN KEY (expense_id) REFERENCES expense(id),
    FOREIGN KEY (tag_id) REFERENCES expense_tag(id),

    UNIQUE (expense_id, tag_id)
);
CREATE INDEX idx_tag_index_uid_tag ON expense_tag_index(uid, tag_id, expense_id);
CREATE INDEX idx_tag_index_expense ON expense_tag_index(expense_id);
```

### 3.6 关键设计规则

1. **金额用整数（分）**：`100.50 元 → 10050`。永远不要用浮点数存金额。
2. **时间戳用 Unix 秒**：所有时间字段都是整数 Unix timestamp。展示时按用户时区转换。
3. **软删除**：核心表用 `deleted` + `deleted_at`。查询时始终加 `deleted = 0` 条件。
4. **用户隔离**：所有数据表带 `uid`，所有查询加 `uid = ?` 条件。
5. **标签多对多**：通过 `expense_tag_index` 关联，创建/修改支出时同步维护关联表。

## 四、API 接口规范

### 4.1 通用约定

- **Base URL**：`/api/v1`（需认证）或 `/api`（认证接口）
- **认证方式**：JWT，通过 Header `Authorization: Bearer <token>` 传递
- **请求格式**：GET 用 query string，POST/PUT 用 `Content-Type: application/json`
- **响应格式**：
  - 成功：直接返回 JSON 数据
  - 错误：`{"detail": "错误描述信息"}`（FastAPI 默认格式）
- **分页**：列表接口使用游标分页，参数 `cursor`（上一页最后一条的 ID）+ `limit`（默认 20）

## 五、后端架构设计

### 5.1 分层架构

```
HTTP 请求
  → FastAPI Router（路由注册，api/__init__.py）
  → Middleware（CORS → JWT 认证 → 全局异常处理）
  → API Handler（api/*.py：参数校验（Pydantic）、调用 Service）
  → Service（services/*.py：业务逻辑、事务管理）
  → Repository / SQLAlchemy Session（数据访问、查询）
  → Database（SQLite）
```

**分层规则**：
- API 层：只做 Pydantic 参数校验和调用 Service，不写业务逻辑
- Service 层：包含所有业务规则（如创建支出时维护标签关联、删除分类时检查是否有支出），管理数据库事务
- 数据访问层：使用 SQLAlchemy Session 直接操作，封装常用查询方法
- Models 层：SQLAlchemy ORM 模型定义
- Schemas 层：Pydantic 请求/响应模型，与 API 一一对应

### 5.2 错误处理

使用 FastAPI 的异常机制 + 全局 exception handler：

```python
# 自定义异常
class AppException(Exception):
    def __init__(self, status_code: int, detail: str):
        self.status_code = status_code
        self.detail = detail

# 常用场景
class NotFoundException(AppException):
    def __init__(self, entity: str):
        super().__init__(404, f"{entity}不存在")

class ForbiddenException(AppException):
    def __init__(self):
        super().__init__(403, "无权操作此资源")

class BadRequestException(AppException):
    def __init__(self, detail: str):
        super().__init__(400, detail)
```

全局 handler 捕获 `AppException` 并返回统一格式的 JSON。

### 5.3 JWT 认证流程

```
1. 注册/登录成功 → 后端生成 JWT（payload: {uid, exp}） → 返回给前端
2. 前端存储 token（localStorage）
3. 每次 API 请求带 Header: Authorization: Bearer <token>
4. JWT 中间件从 Header 解析 token → 验证签名和过期时间 → 注入当前 uid
5. 所有 handler 通过依赖注入获取当前 uid，确保用户隔离
```

JWT 过期时间默认 7 天，支持通过 refresh 接口续期。

## 六、前端架构设计

### 6.1 移动端 PWA 架构

前端为纯移动端 SPA + PWA，通过 `manifest.json` + Service Worker 支持添加到手机桌面。移动优先设计，无需设备检测分流。

核心页面：
```
/login              →  登录
/register           →  注册
/                   →  首页概览（今日/周/月/年汇总）
/expenses           →  支出列表（筛选、分页）
/expenses/add       →  记账表单
/expenses/:id/edit  →  编辑支出
/categories         →  分类管理
/tags               →  标签管理
/statistics         →  统计分析（多维度图表）
/profile            →  个人信息/修改密码
```

### 6.2 前端分层

```
Pages（页面组件，纯模板 + 调用 Store）
  → Stores（状态管理：状态缓存、API 调用、数据转换）
    → Lib（基础设施：HTTP 客户端封装、Token 管理、路由守卫）
      → Core（纯工具函数：金额格式化、日期处理、分类图标映射等）
```

关键设计：
- 页面组件只负责 UI 渲染和事件分发，不直接调 API
- API 调用统一收拢在 Store 的 actions 中
- 金额在前端展示时除以 100 格式化为「元」，输入时乘以 100 转为「分」

### 6.3 前端框架选型

已确定使用 **Vue 3 + Vant 4**：

| 层 | 选型 | 说明 |
|----|------|------|
| 框架 | Vue 3.5+ | Composition API + `<script setup>` |
| 构建 | Vite 6+ | 快速 HMR |
| 语言 | TypeScript 5.x | 类型安全 |
| UI 组件 | Vant 4 | 轻量移动端组件库（~50KB gzipped），按需引入 |
| 路由 | Vue Router 4 | SPA 路由 |
| 状态管理 | Pinia | Vue 3 官方推荐 |
| HTTP | Axios | 拦截器处理 JWT |
| 图表 | ECharts 5 | 饼图、柱状图、折线图 |
| PWA | vite-plugin-pwa | manifest + Service Worker |

选择 Vant 4 的理由：轻量（~50KB gzipped）、移动端组件刚好够用（NavBar、Tabbar、Form、Field、Picker、DatetimePicker、Dialog 等）、按需引入 Tree-shaking、中文文档完善。

## 七、统计系统设计

### 7.1 统计维度

所有统计支持三个维度的筛选，可任意组合：

- **时间维度**：`start_time` ~ `end_time`（Unix 时间戳范围）
- **分类维度**：`category_ids`（逗号分隔的分类 ID 列表）
- **标签维度**：`tag_ids`（逗号分隔的标签 ID 列表）

统计计算时：
1. 先按筛选条件从 expense 表 + expense_tag_index 表联表查出符合条件的支出
2. 按目标维度（分类/标签/月份）做 GROUP BY 聚合
3. 计算每组的总额、占比、笔数

### 7.2 统计 SQL 示例

按分类统计（带时间范围和标签筛选）：

```sql
SELECT e.category_id,
       SUM(e.amount) AS total_amount,
       COUNT(*) AS transaction_count
FROM expense e
JOIN expense_tag_index eti ON e.id = eti.expense_id
WHERE e.uid = ? AND e.deleted = 0
  AND e.transaction_time BETWEEN ? AND ?
  AND eti.tag_id IN (?, ?)          -- 标签筛选（可选）
GROUP BY e.category_id
ORDER BY total_amount DESC;
```

### 7.3 图表类型

| 统计接口 | 图表类型 | 说明 |
|---------|---------|------|
| overview | 卡片数字 | 四个时间窗口的支出总额 |
| by_category | 饼图 | 各分类占比，可点击钻取详情 |
| by_tag | 饼图 | 各标签占比 |
| monthly | 柱状图 + 折线图 | 月度趋势，柱状图按分类堆叠 |

## 八、构建和部署

### 8.1 开发环境启动

```bash
# 后端（使用 Anaconda 管理 Python 环境）
conda create -n myaccounting python=3.12 -y
conda activate myaccounting
cd backend
pip install -r requirements.txt
uvicorn main:app --reload --host 0.0.0.0 --port 8080

# 前端
cd frontend
npm install
npm run dev
```

### 8.2 生产部署

**方式一：直接运行**

```bash
pip install -r requirements.txt
uvicorn main:app --host 0.0.0.0 --port 8080
```

**方式二：Docker Compose（推荐）**

```bash
# 修改 docker-compose.yml 中的 JWT 密钥后启动
docker compose up -d
```

**方式三：Docker 手动**

```bash
docker build -t ezexpense .
docker run -d \
  -p 8080:8080 \
  -v /path/to/data:/app/data \
  -e APP_SECURITY_JWT_SECRET=your-secret \
  ezexpense:latest
```

### 8.3 配置覆盖

所有 `config.yaml` 中的配置可通过环境变量覆盖：

```bash
APP_SERVER_PORT=9090
APP_DATABASE_PATH=/data/my_expense.db
APP_SECURITY_JWT_SECRET=your-secret-key
```

## 十、设计原则总结

1. **金额整数存储**：全部以「分」为单位，int64，避免浮点精度问题
2. **时间统一 Unix 秒**：存储和计算用时间戳，展示时按用户时区格式化
3. **软删除统一模式**：核心表用 `deleted + deleted_at`，查询始终加 `deleted = 0`
4. **用户数据严格隔离**：所有表带 `uid`，所有查询带 `uid = ?`
5. **标签多对多**：通过关联表维护，修改支出时原子性更新
6. **事务保护**：涉及多表写入的操作（创建支出 + 标签关联）使用数据库事务
7. **分层清晰**：API → Service → Repository，职责不越界
8. **配置环境变量覆盖**：方便 Docker 部署和不同环境切换
