# 认证系统从JWT改为PIN码

## Status: IMPLEMENTED (2026-06-25)

## 背景

当前系统使用JWT（HS256）+ bcrypt密码认证，支持注册、登录、忘记密码、刷新令牌等功能。产品定位为内网家庭两人使用，JWT的复杂性（令牌刷新、无状态设计、密码策略）与使用场景不匹配。

## 决策

- 认证方式：从用户名+密码改为用户名+PIN码（纯数字4-6位）
- 会话管理：从JWT无状态令牌改为服务端Cookie Session（24小时过期）
- 用户管理：移除注册功能，通过数据库迁移预置两个用户
- PIN码重置：移除邮箱验证，改为固定安全问题重置
- 存储方式：PIN码哈希复用现有password字段，移除email字段

## 理由

1. **简化用户体验**：PIN码输入比密码快，适合手机端频繁使用
2. **降低系统复杂度**：移除JWT刷新机制、令牌管理、密码策略等复杂逻辑
3. **匹配使用场景**：内网家庭环境，安全性要求较低，简单性优先
4. **减少维护成本**：无需处理JWT过期、令牌泄露等安全问题

## 备选方案

- **保持JWT + 简化密码策略**：仍需处理令牌刷新、无状态设计等复杂性
- **HTTP Basic Auth**：浏览器原生弹窗，用户体验差，无法自定义界面
- **IP白名单**：无法区分用户，失去数据隔离能力
- **完全移除认证**：失去用户隔离，无法保护个人数据

## 实施计划

1. 修改User模型：移除email字段，password字段存储PIN码哈希
2. 修改认证服务：移除注册、忘记密码（邮箱验证），改为PIN码验证+安全问题重置
3. 修改会话管理：移除JWT中间件，实现Cookie Session中间件
4. 修改前端：移除注册页面，改为PIN码输入界面
5. 数据库迁移：预置两个用户，迁移现有数据
6. 修改API端点：移除注册、刷新令牌端点，添加安全问题验证端点

## 已澄清问题

- 安全问题：固定问题「小1是谁？」，所有用户共用
- 默认PIN码：预置用户使用简单默认值（如1234），首次登录后强制修改
- 会话存储：内存存储，重启后会话丢失
- 昵称修改：保留用户昵称修改功能
